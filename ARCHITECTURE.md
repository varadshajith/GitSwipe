# GitSwipe System Architecture

This document provides a comprehensive technical overview of the GitSwipe platform, detailing its data flows, agentic orchestration, and infrastructure components.

## 1. System Design Objectives

GitSwipe is designed as an agentic system that transforms raw external data from the GitHub API into structured, personalized developer insights. The architecture prioritizes low-latency inference, asynchronous processing, and modular agent design.

### Core Lifecycle Data Flow

1.  **Ingestion Layer (Scout Agent)**: Orchestrates distributed search processes to identify potential repository candidates across multiple dimensions.
2.  **Scoring Layer (Ranker Agent)**: Applies a weighted scoring algorithm (hybrid heuristic and LLM) to evaluate candidates against specific developer profiles.
3.  **Synthesis Layer (Explainer Agent)**: Processes repository metadata and source code into multi-tier summaries (concise card metadata and extensive technical deep-dives).
4.  **Interaction Layer (Guide Agent)**: Facilitates contextual repository exploration via Retrieval-Augmented Generation (RAG).

---

## 2. Multi-Agent Orchestration

The system utilizes **LangGraph** for managing agent state and executing complex, non-linear workflows.

### 2.1 Scout Agent (Discovery Engine)
The Scout node initiates parallel sub-processes across several search vectors to maximize discovery breadth.

| Vector | Specification |
| :--- | :--- |
| **Topic-centric** | Targeted search via GitHub API based on user-defined technical interest topics. |
| **Organizational** | Analysis of repositories within organizations previously explored or starred by the user. |
| **Technographic** | Identification of projects based on shared dependency patterns and tech-stack alignment. |
| **Emergent (Low-Star)** | Discovery of high-activity projects with low star counts to identify early-stage opportunities. |
| **Contributor-focused** | Prioritization of repositories with significant active `good-first-issue` labels. |
| **Momentum-based** | STAR velocity analysis to identify repositories with high growth trajectories. |

**Execution Pattern**: Vectors execute concurrently within the LangGraph graph, aggregating result sets into a unified `RepositoryCandidate` collection before deduplication.

### 2.2 Ranker Agent (Prioritization Engine)
Translates high-volume candidate lists into highly relevant personalized feeds.

-   **Model Selection**: `llama-3.1-8b-instant` for high-throughput, low-latency categorical scoring.
-   **Weighted Formula**:
    -   `Heuristic_Score (30%)`: Calculated via project metadata (growth rate, commit frequency, language overlap).
    -   `Contextual_Score (70%)`: Generated by the LLM by evaluating the repository README against the `PreferenceProfile`.

### 2.3 Explainer Agent (Synthesis Engine)
The Explainer utilizes a Map-Reduce strategy to handle large repository documentation files.

-   **Processing Pipeline**:
    1.  **Map Phase**: Parallel summarization of 1000-token README segments.
    2.  **Reduce Phase**: Synthesis of segment summaries with file structure analysis to produce structured output.
-   **Output Artifacts**:
    -   **Card Summary**: Fixed-length (200 chars) summary for feed consumption.
    -   **Technical Deep-Dive**: Structured Markdown providing architectural overviews, core logic locations, and contribution pathways.

### 2.4 Guide Agent (Interactive Consultation)
A RAG-enabled agent for real-time repository navigation.

-   **Backend Model**: `llama-3.3-70b-versatile`.
-   **Available Toolset**:
    -   `fetch_file`: Retrieval of full source code for specific files.
    -   `search_issues`: Contextual querying of project issue history.
    -   `find_similar_repos`: Semantic search within the vector database to identify related projects.
-   **Memory Management**:
    -   **Ephemeral (Session)**: Contextual thread history.
    -   **Persistent (User)**: Long-term storage of user expertise levels to adjust technical explanation depth.

## 3. LangGraph Execution Flows

### 3.1 Feed Generation Pipeline
1.  **Trigger**: Scheduled job or manual refresh.
2.  **Scout Node**: Conducts parallel search across all vectors.
3.  **Filtration Node**: Deduplicates results and applies initial star-count and recency thresholds.
4.  **Ranker Node**: Evaluates candidate relevance via LLM.
5.  **Explainer Node**: Generates feed-ready summaries.
6.  **Persistence**: Stores finalized feed in PostgreSQL.

### 3.2 Guide Interaction Pipeline
1.  **Input**: User query initiated from a specific repository view.
2.  **Retrieval**: Identification of relevant code chunks via vector search.
3.  **Inference**: Guide agent processes query and context.
4.  **Tool Execution**: Agent optionally invokes tools (e.g., `fetch_file`) to expand context.
5.  **Return**: Streamed response with Markdown formatting and source citations.

## 4. Preference Profile Specification

The profile is stored as a structured JSON object in PostgreSQL, serving as the primary context for the Ranker and Scout agents.

```json
{
  "user_id": "uuid",
  "technical_stack": { "python": 0.9, "typescript": 0.6, "go": 0.4 },
  "interest_domains": ["distributed-systems", "compiler-design", "ai-infrastructure"],
  "filtering_constraints": {
    "min_stars": 250,
    "max_stars": 10000,
    "license": ["MIT", "Apache-2.0"]
  },
  "engagement_vectors": {
    "infrastructure": +12,
    "ui-components": -5
  }
}
```

---

## 5. RAG Infrastructure and Vector Database

-   **Embedding Model**: `all-MiniLM-L6-v2` (Deployed locally for cost efficiency and low latency).
-   **Segmentation Strategy**: 1000-token windows with a 200-token sliding overlap.
-   **Database Technology**: ChromaDB (V1) providing filesystem-based vector persistence.
-   **Retrieval Algorithm**: Top-k (k=5) Cosine Similarity.

---

## 6. Background Processing and Job Scheduling

All background tasks are orchestrated via `APScheduler`.

| Task | Interval | Description |
| :--- | :--- | :--- |
| **Feed Ingestion** | 12 Hours | Executes the full Scout-Ranker-Explainer pipeline. |
| **Weight Synchronization** | Event-driven | Updates `PreferenceProfile` weights every 5 user interactions. |
| **Metadata Refresh** | 24 Hours | Synchronizes star counts and recent commit data for saved repositories. |

## 7. API Specification (Core Endpoints)

### Feed Management
- `GET /api/v1/feed`: Retrieves current personalized repository feed.
- `POST /api/v1/feed/action`: Records user interaction (Save, Skip, Explore).

### Discovery Tools
- `GET /api/v1/exploration/{repo_id}`: Retrieves comprehensive repository deep-dive metadata.
- `POST /api/v1/tools/generate-pr`: Processes a git diff to generate formal PR documentation.

### Agentic Communication
- `WS /api/v1/chat/{repo_id}`: Persistent WebSocket connection for the Guide agent.
